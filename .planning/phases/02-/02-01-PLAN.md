## Objective

Create and secure the `user_access` table with proper indexes and RLS policies to support payment-based access control. This table stores one-time and subscription access granted through MercadoPago payments, complementing the `user_manual_access` table for admin-granted access.

**Purpose:** Ensure database layer is secure, performant, and ready for Phase 3 admin panel development.

**Output:** Complete user_access table with indexes, RLS policies, and verification queries.

Create migration file 007_create_user_access_table.sql with:

1. **Table schema** (based on usage in payments.ts and mercadopago.ts):
   - id: UUID PRIMARY KEY DEFAULT gen_random_uuid()
   - user_id: UUID REFERENCES auth.users(id) ON DELETE CASCADE NOT NULL
   - access_type: TEXT CHECK (access_type IN ('subscription', 'one_time')) NOT NULL
   - payment_id: UUID REFERENCES payments(id) (nullable for future flexibility)
   - expires_at: TIMESTAMPTZ NOT NULL
   - is_active: BOOLEAN DEFAULT true
   - created_at: TIMESTAMPTZ DEFAULT now()
   - updated_at: TIMESTAMPTZ DEFAULT now()
   - UNIQUE constraint on (user_id, access_type) to prevent duplicate active access

2. **Enable RLS:** ALTER TABLE user_access ENABLE ROW LEVEL SECURITY

3. **Initial RLS policies:**
   - Service role full access: USING (auth.role() = 'service_role')
   - Users can read own access: USING (auth.uid() = user_id)

4. **Verification query** at end to confirm table creation

**DO NOT create indexes yet** - Task 2 will handle them separately.

<task type="auto">
  <name>Task 2: Add performance indexes and explicit deny policies</name>
  <files>supabase/migrations/008_add_user_access_indexes.sql</files>
  <action>
Create migration file 008_add_user_access_indexes.sql with:

1. **Composite index for active access queries** (99.94% performance improvement per research):
   ```sql
   CREATE INDEX idx_user_access_active
     ON user_access(user_id, is_active, expires_at);
   ```

2. **Explicit deny policies** (SEC-04 requirement):
   - "Users cannot update own access" - USING (false)
   - "Users cannot delete own access" - USING (false)
   - "Service role can update access" - USING (auth.role() = 'service_role')
   - "Service role can delete access" - USING (auth.role() = 'service_role')

3. **Rationale:** Explicit deny policies prevent users from modifying their own access records, even if client-side bugs try to do so. Service role (server-side) bypasses RLS entirely for admin operations.

4. **Verification queries:**
   - Check indexes exist: SELECT * FROM pg_indexes WHERE tablename = 'user_access'
   - Check policies: SELECT * FROM pg_policies WHERE tablename = 'user_access'
  </action>

  </verify>
  <done>Migration file exists with composite index and explicit deny RLS policies</done>
</task>
